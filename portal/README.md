# velocity-portal

React + Vite SPA for managing Velocity objects.

## How it's served

The portal is **bundled into `velocity-api`** via `rust-embed` â€” no
separate container, no nginx. The root `Dockerfile` runs `npm run build`
in a node stage, drops `dist/` into `crates/velocity-api/static/`, and
`rust-embed` picks it up at compile time. At runtime, the API's
`static_files` module mounts the embedded asset set as the router's
fallback so `/` serves `index.html`, `/assets/*.js` serves the bundle,
and any unknown deep link falls back to `index.html` for the SPA's
client-side router.

Runtime knobs land at `GET /config.json`, generated by the API from
three env vars (`VELOCITY_API_PORTAL_DEFAULT_AUTH_STRATEGY`,
`VELOCITY_API_PORTAL_GRAFANA_URL`, `VELOCITY_API_PORTAL_ENVIRONMENT`).

## Layout

- Dark theme + amber accent, monospace.
- Three-panel shell (sidebar / main / page-specific right panes).
- Routes mirror the `org/app/domain/object/version` addressing scheme.

## Backend coupling

All API calls are relative; the portal and API share an origin by
construction. In production a single Ingress (`api.ingress` in the
chart) routes the hostname to the api Service; in dev Vite proxies
`/api`, `/auth`, `/version`, `/healthz`, `/readyz` to
`$VELOCITY_API_BASE` (default `http://127.0.0.1:8080`).

CRD management views (AuthStrategy / RoleBinding / ApiKey /
LogFilterPolicy / LogRoutingPolicy / SchemaDefinition) are
**create-only** in the portal: they generate a YAML manifest with a
live preview, then offer "Copy YAML" / "Copy `velocity apply` command"
buttons. Listing existing CRDs requires either `velocity get` or
`kubectl get` â€” there is no admin API for CRD reads today.

## Local development

```sh
cd portal
npm install
VELOCITY_API_BASE=http://localhost:8080 npm run dev
```

Vite serves the SPA on `:5173` with HMR; the API serves on `:8080`.

## Tests

```sh
npm run test
```

Vitest + jsdom. Covers the API client (fetch wrapper, error mapping,
401 event dispatch, 204 handling) and the YAML serialization shape used
by the visual editors.

preview a release build locally without a cluster:

```sh

# open http://localhost:8080/
```

## Helm

Configure via the `api.portal` block in the umbrella chart values:

```yaml
api:
  portal:
    defaultAuthStrategy: "velocity/portal-oidc"
    grafanaUrl: "https://grafana.example.com"
    environment: "PROD"
  ingress:
    enabled: true
    className: "nginx"
    hosts:
      - host: velocity.example.com
        path: /
        pathType: Prefix
```

The API binary reads these via env vars and emits them at `/config.json`
on every request, so no ConfigMap mount is needed.
